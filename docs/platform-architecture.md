# Toss 미니앱 구조 분석 — 스콧 & 노아 회의록

## 1. Toss 미니앱이란?

Toss 앱 내부에서 실행되는 **웹뷰 기반 미니 프로그램**. 사용자는 Toss 앱을 따로 설치할 필요 없이 Toss 앱 내 '미니앱' 탭에서 바로 실행.

### 제약 조건

| 항목 | 제약 | 영향 |
|------|------|------|
| **UI** | Toss 디자인 시스템 준수, 헤더/내비 고정 | 자유도 제한, Toss 느낌 필수 |
| **웹뷰** | 모든 UI는 HTML/CSS/JS, 네이티브 API 제한 | 카메라/푸시 등 네이티브 기능 불가 |
| **로그인** | Toss 계정 OAuth 자동 연동 | 별도 로그인 UX 불필요, 토큰 활용 가능 |
| **저장소** | localStorage 제한 (~5MB), 쿠키 제한 | DB는 서버 필요, 로컬 캐싱만 |
| **네트워크** | CORS 제한, HTTPS 필수 | API 서버는 HTTPS + 허용 origin 등록 |
| **결제** | Toss Pay 연동 가능 | 간식박스 유료화 시 활용 가능 (Post-MVP) |
| **공유** | Toss 공유 API 호출 가능 | 초대코드 공유 시 Toss 친구탭 활용 |
| **푸시** | Toss 알림 채널 활용 가능 | 단, 미니앱 자체 푸시는 불가, Toss 채널→유저 |
| **용량** | 최초 로딩 2MB 이내 권장 | 이미지/번들 최적화 필수 |
| **심사** | Toss 미니앱 센터 심사 필수 | 기능/UI/보안 모두 심사 대상 |

## 2. 구조 회의 — 스콧 & 노아

**스콧**: "Toss 미니앱 구조를 다시 파악해야 한다. 우리가 만든 React+Vite 프론트엔드는 맞지만, Toss 웹뷰 환경에 맞춰 최적화해야 한다."

**노아**: "동의. 가장 큰 변화는 (1) Toss 로그인 연동 (2) 서버 비용 0원 전략 유지 (3) Toss 디자인 시스템 적용이다."

**스콧**: "서버는 어떻게 할까? 현재 Express 서버가 있는데, 비용 0원 기조를 유지하려면?"

**노아**: "2가지 옵션이 있다. **옵션A**: 서버리스 — 모든 로직을 프론트엔드로, localStorage에 데이터 저장. **옵션B**: 경량 API — Vercel/Cloudflare Workers 무료 티어 사용."

**스콧**: "MVP는 옵션A로 가자. 서버 비용 0원, 유저 데이터는 localStorage, 문제 데이터는 JS 번들에 포함. 단, 백엔드 코드는 구조적으로 유지하되 실제 배포 시 선택 가능하게."

**노아**: "좋다. 그러면 아키텍처를 두 가지 트랙으로 나누자. (1) **로컬 트랙**: localStorage 전용, 서버 필요 없음 (2) **API 트랙**: Express 서버, 랭킹 등 고급 기능."

**스콧**: "MVP는 로컬 트랙으로 출시. 이후 업데이트에서 API 트랙 추가. 이 결정을 문서화하고 프론트엔드/백엔드팀에 전달하자."

**노아**: "Toss 디자인 시스템 적용 범위도 정리해야 한다. 폰트는 Toss Face(토스 페이스), 컬러는 Toss 블루(#3182f6), 라운딩은 12px 고정 등."

**스콧**: "그래. 전체 구조 파악 문서를 v4로 작성하고, 각 팀에 전달. Step 2(프론트)부터는 Toss 웹뷰 환경에 맞춰 개발하도록."

## 3. 결정된 아키텍처

```
[사용자] → Toss 앱 → 미니앱 런처 → 웹뷰 → KICK QUIZ 프론트엔드
                                                           │
                                     ┌─────────────────────┤
                                     ▼                     ▼
                               localStorage            (선택) API 서버
                               (문제/점수/스트릭)       (랭킹/크롤링)

MVP: 로컬 트랙만 사용. API 서버는 Post-MVP.
```

### 데이터 흐름 (로컬 트랙)

| 데이터 | 저장 위치 | 용량 | 비고 |
|--------|----------|------|------|
| 문제 DB | JS 번들 | ~200KB | 245문항 포함 |
| 유저 점수/P | localStorage | ~10KB | key: `kq_user` |
| 퀴즈 세션 | localStorage | ~2KB | key: `kq_session` |
| 스트릭 | localStorage | ~1KB | key: `kq_streak` |
| 쿠폰함 | localStorage | ~2KB | key: `kq_coupons` |
| 설정 | localStorage | ~1KB | key: `kq_settings` |
| 초대코드 | localStorage | ~1KB | key: `kq_invite` |

### 보안 고려사항 (Toss 웹뷰)

| 위협 | 대응 |
|------|------|
| localStorage 변조 | 중요 데이터는 검증 로직 추가 (예: 점수 증가만 허용) |
| XSS | React 기본 방어 + innerHTML 사용 금지 |
| HTTPS | Toss 웹뷰는 HTTPS 강제 |
| CORS | API 서버 도입 시 Toss origin만 허용 |
